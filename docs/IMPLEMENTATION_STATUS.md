# Implementation Status and Next Steps

## Implemented Features
- **Shared foundation**: Tenants own households, properties, events, and users to keep families isolated.
- **Family directory context**: Households group members; member records store names, birth dates, dietary notes/tags.
- **Gathering planning context**: Events tie to properties and manage meal templates/plans, task templates/plans, accommodations, and member intents for attendance, meals, stays, and tasks.
- **Detailed implementation**: Multi-tenant backbone with auditing, households and members (including dietary metadata), relationship graph, contact/addresses/dietary profiles, event scaffolding with meal templates (`MealTemplate` → `MealPlan` → `MealAttendance` + `MealIntent`) and task templates (`TaskTemplate` → `TaskPlan` → `TaskIntent`), daily attendance, accommodation intents with arbitration metadata, equipment inventory, and custom extensible attributes embedded on all major entities. Tenant identifiers on dependent tables enforce tenant-aware filters across the graph.
- **Family Directory CRUD API**: Full CRUD controllers and services for all Family Directory sub-entities:
  - **Addresses** (`/members/{memberId}/addresses`): CRUD with automatic IsPrimary flag management (setting one primary unsets others).
  - **Contact Methods** (`/members/{memberId}/contacts`): CRUD for email/phone/other with same IsPrimary logic.
  - **Member Relationships** (`/members/{memberId}/relationships`): CRUD with cross-member validation (related member must exist in the same tenant, can span households), self-relationship guard, and duplicate relationship check.
  - **Dietary Profiles** (`/members/{memberId}/dietary-profile`): GET/PUT(upsert)/DELETE for one-to-one profiles, plus a tenant-level list endpoint (`/tenants/{tenantId}/dietary-profiles?memberIds=`) for aggregating dietary needs across attending members for meal planning.
- **JWT Bearer authentication (Entra External ID / Azure AD B2C)**: Authentication uses standard JWT Bearer tokens validated against a Microsoft external identity provider (Entra External ID or Azure AD B2C). The `ExternalIdentity` configuration section supports both providers — Entra External ID uses `ciamlogin.com` endpoints while B2C uses `b2clogin.com`; the authority URL conditionally includes the policy segment only when `SignUpSignInPolicyId` is configured. The API validates issuer, audience, lifetime, and signing keys via OIDC metadata discovery. Token revocation is supported via a `RevokedTokens` table checked in the `OnTokenValidated` event. `HttpContextCurrentUserContext` resolves the external identity provider subject (`sub` claim) to an internal `User.Id` via a DB lookup, with per-request caching in `HttpContext.Items`. (Note: Consider PASETO migration when ecosystem support improves.)
- **Rate limiting**: Sliding-window rate limiter configured in the API pipeline to enforce request throttling per the secure-defaults principle.
- **Tenant-scoped authorization**: `RequireTenantAccessAttribute` authorization filter that extracts the tenant from the route, verifies the user's `TenantUser` membership, and enforces a role hierarchy (Owner > Manager > Member > Guest). Applied to tenant-aware controllers.
- **Always Encrypted data protection**: Shifted from application-level encryption to SQL Server Always Encrypted, delegating column-level encryption of sensitive fields to the database driver rather than custom application code.
- **Project restructure**: Migrated data projects from `packages/db/` to `src/Gatherstead.Data/` and `src/Gatherstead.Data.Setup/`, aligning namespaces under `Gatherstead.Data`.
- **Framework upgrade**: Updated all projects to target .NET 10 (`net10.0`) with current package references.
- **Composable soft-delete query filters**: Global query filters in `GathersteadDbContext` now conditionally apply the soft-delete clause via a `_includeDeleted` field, while tenant isolation remains unconditional. List/read endpoints accept `?includeDeleted=true`, RBAC-gated to `TenantRole.Manager+` in `RequireTenantAccessAttribute`. The authorization decision flows through `IIncludeDeletedContext` → `HttpContext.Items`, ensuring the raw query parameter cannot bypass role checks. Lower-role users' flag is silently ignored.
- **Cross-tenant write prevention**: The `AuditingSaveChangesInterceptor` validates that every new entity's `TenantId` matches the current tenant context before saving, throwing an `InvalidOperationException` on mismatch. This provides a defense-in-depth backstop against accidental or malicious cross-tenant writes.
- **Temporal history retention**: `Gatherstead.Data.Setup` configures a 1-year retention policy on all temporal (system-versioned) tables, keeping history bounded and storage predictable.
- **Integration test suite**: A dedicated `Gatherstead.Api.Tests` project covers the authentication pipeline (JWT Bearer validation, invalid/expired/wrong-key tokens), CORS policy, rate limiting, security headers, tenant access authorization, token revocation, and context propagation (`ICurrentTenantContext`, `ICurrentUserContext`, `IIncludeDeletedContext`).
- **API contract unification and TypeScript auto-generation**: All API contract domains migrated from per-type individual files (`CreateXRequest.cs`, `UpdateXRequest.cs`, `XResponse.cs`) to a single bundled `*Contracts.cs` file per domain — keeping all of a domain's wire types co-located and matching the established pattern of newer domains. A global `JsonStringEnumConverter` was added to `AddControllers()` in `Program.cs` (and `UseInlineDefinitionsForEnums()` to `AddSwaggerGen()`) so non-flags string enums emit correctly in the OpenAPI spec; `[Flags]` enums (`MealTypeFlags`, `TaskTimeSlotFlags`) override with `[JsonConverter(typeof(JsonNumberEnumConverter<T>))]` per-field. Frontend types are now fully auto-generated: `scripts/generate-openapi.sh` orchestrates Swashbuckle CLI → `openapi-typescript` → `tools/FlagsCodegen` (C# enum reflection for bit values); the committed `openapi.json`, `generated/api.d.ts`, and `generated/flags.gen.ts` mean the frontend build never needs dotnet. `app/repositories/types.ts` is a pure barrel with no hand-maintained definitions; `typeUtils.ts` holds the three flag utility functions. A `check-openapi-freshness` CI job diffs a freshly-generated spec against the committed file to prevent silent drift.
- **Supply-chain & dependency policy**: Committed a tiered stand-off policy for dependency updates ([SECURITY-DEPS.md](SECURITY-DEPS.md)) with CISA-KEV-based exception criteria and an emergency-patch runbook. Wired Dependabot ([.github/dependabot.yml](../.github/dependabot.yml)) for weekly grouped PRs across nuget/npm/github-actions. Added CI gates to [.github/workflows/build-and-test.yml](../.github/workflows/build-and-test.yml): `audit-nuget`, `audit-pnpm` (`--audit-level=high`), and `dependency-review`. All .NET projects (API, Data, Data.Setup, Api.Tests) set `RestorePackagesWithLockFile=true` with committed `packages.lock.json`; CI builds use `--locked-mode` / `--frozen-lockfile` for integrity.
- **Four-tier RBAC**: Fine-grained write authorization enforced at the service layer via `IMemberAuthorizationService`. Tenant roles (Owner=0 > Manager=1 > Coordinator=2 > Member=3 > Guest=4) compose with Household roles (Manager=0, Member=1) to form a layered permission model:
  - **`CanManageTenantAsync`** (Manager+): structural resources — Properties, Accommodations, Households, Users.
  - **`CanManageEventAsync`** (Coordinator+): event resources — Events, MealTemplates, TaskTemplates, MealPlans, TaskPlans. Implemented via `AuthorizeEventManageAsync` in `ServiceGuards`.
  - **`CanAssignIntentForMemberAsync`** (Coordinator+, Self, or Household Manager): intent/attendance records — EventAttendance, MealAttendance, MealIntent, TaskIntent, AccommodationIntent. Implemented via `AuthorizeIntentAssignAsync` in `ServiceGuards`.
  - **`CanEditMemberAsync`** (Manager+, Self, or Household Manager): member profile data — name, contact info, addresses, dietary notes, attributes, relationships.
  - **`CanManageHouseholdAsync`** (Manager+ or Household Manager): household add/remove/edit.
  - `HouseholdUser` join entity ties a `User` to a `Household` with a `HouseholdRole` (`Manager=0`, `Member=1`), mirroring `TenantUser`. PK `(HouseholdId, UserId)`; carries `TenantId` for global-filter isolation. Household management authority derives from this record, not from any field on `HouseholdMember`.
  - `TenantUser.LinkedMemberId` (nullable FK → `HouseholdMember`) identifies a user's self-profile within a given tenant. The FK lives on `TenantUser` (not `HouseholdMember`) so users who belong to multiple tenants each have a distinct self-profile per tenant. The reverse lookup (`LinkedUserId`) is intentionally absent from `HouseholdMemberDto` — the natural direction is user→member. Managing the link is a dedicated operation: `PUT api/tenants/{tenantId}/households/{householdId}/members/{memberId}/linked-user` (`SetLinkedUserRequest.UserId`; null to unlink). Requires Tenant Owner/Manager or Household Manager and is validated to prevent silent overwrite of an existing link.
  - `User` carries `HouseholdUsers` navigation collection; `Household` carries a `HouseholdUsers` collection. `User.HouseholdMembers` (old collection nav) was removed in favour of `TenantUser.LinkedMemberId`.
  - **Sensitive read scope**: `GetSensitiveReadScopeAsync(tenantId)` returns a `SensitiveReadScope` value — `Global` (App Admin or TenantRole ≤ Member), `ForHouseholds(ids)` (Guest with `HouseholdUser` entries — scoped to those households), or `None` (Guest with no `HouseholdUser` entries). `HouseholdMemberService` resolves the scope once per request and nulls `BirthDate`, `DietaryNotes`, and `DietaryTags` in the DTO when the caller lacks access for that household. Sub-entity read endpoints (Addresses, ContactMethods, MemberRelationships, DietaryProfile GET) call `AuthorizeSensitiveReadAsync` or `AuthorizeGlobalSensitiveReadAsync` and return 403 when access is denied. `HouseholdMemberAttribute` visibility additionally applies the `HouseholdMinRole` bypass described in the Custom extensible attributes section.
  - All write paths cache `TenantUserInfo` (role + `LinkedMemberId`) and household-user roles in `HttpContext.Items` (two keys) to avoid redundant DB queries within a single request. The Self check reads `LinkedMemberId` directly from the cached `TenantUserInfo` — no separate DB query.
  - Role assignment is Tenant Manager+ only; a user may not grant roles more privileged than their own (`ServiceGuards.RequireNonEscalatingRole`). Household Managers cannot independently assign roles.
  - A `HasOtherOwnerAsync` helper in `TenantService` enforces that a tenant always retains at least one Owner — must be called before any TenantUser demotion or removal operation.
  - `CreateHouseholdMemberRequest` and `UpdateHouseholdMemberRequest` contain only member profile fields (name, age, dietary data). The `TenantUser.LinkedMemberId` link is managed separately via the `PUT .../linked-user` endpoint, keeping `TenantUser` mutations out of `HouseholdMember` create/update flows.
- **Tenant Owner-only mutations**: Tenant update and delete operations are restricted to the `TenantRole.Owner` role.
- **App Admin role**: Platform-level `IsAppAdmin` flag on the `User` entity grants full administrative privileges. App Admins bypass all tenant-level (`RequireTenantAccessAttribute`) and resource-level (`IMemberAuthorizationService`) authorization checks. Key behaviors:
  - Tenant creation (`POST /api/tenants`) is restricted exclusively to App Admins via `RequireAppAdminAttribute`. The request specifies an `OwnerUserId` to designate the initial tenant Owner.
  - `GET /api/tenants` returns all tenants for App Admins, while regular users see only tenants they belong to.
  - App Admin status is resolved per-request via `IAppAdminContext` with `HttpContext.Items` caching.
  - The first App Admin must be set via direct SQL (`UPDATE Users SET IsAppAdmin = 1 WHERE ...`).
- **Localization (i18n)**: `@nuxtjs/i18n` v10 integrated with lazy-loaded locale files (English, Spanish), `prefix_except_default` URL strategy, cookie persistence, and browser language detection. See [LOCALIZATION.md](agents/plans/LOCALIZATION.md).
- **Demo site**: `demoMode` runtime config flag implemented; `useAuth`, `useTenants`, and route middleware all branch on it, stubbing auth and tenant data with in-memory values. Full localStorage persistence layer is implemented via `DemoStore` (reactive `ref()` arrays persisted to `localStorage`). Demo entities use `crypto.randomUUID()` for IDs. `seedDemoData.ts` programmatically creates all seed data through repository CRUD methods in dependency order, exercising the same code paths as real usage. Demo repositories auto-generate `MealPlan`/`TaskPlan` records per `(day × mealType/timeSlot)` when a template is created (matching the live `PlanSyncService` behavior), respecting template-level `startDate`/`endDate` when set. Seed data includes full event attendance for all three households across all event days, multi-day meal attendance for each member, and task templates demonstrating `startDate`/`endDate` scoping (e.g., a "Set Up" task pinned to day 1 and a "Tear Down" task pinned to the last day). A "Reset Demo Data" button in the demo limits modal calls `clearDemoStore()` and re-runs `seedDemoData`. See [DEMO_SITE.md](agents/plans/DEMO_SITE.md).
- **Hybrid rendering**: Nuxt `routeRules` configured — public root is prerendered for SEO; authenticated tenant routes (`/tenants/**`) are client-only SPA.
- **Event plan auto-generation**: `PlanGenerator` computes `TaskPlan` and `MealPlan` diffs when an event's date range changes — adding plans for new days, restoring soft-deleted non-exception plans, and pruning empty plans for removed days. `EventService.SyncPlansAsync()` is the call site, covered by unit tests. Both `GenerateTaskPlanDiff` and `GenerateMealPlanDiff` accept optional `startDate`/`endDate` parameters; when a template has its own sub-range set, the generator uses those as the effective window instead of the full event range.
- **Guard composition pattern**: Static `ServiceGuards` class provides composable async guards that short-circuit on failure by mutating the shared response object. Full set: `RequireRequest`, `LoadOrNotFoundAsync`, `AuthorizeTenantManageAsync`, `AuthorizeEventManageAsync`, `AuthorizeMemberEditAsync`, `AuthorizeIntentAssignAsync`, `AuthorizeHouseholdManageAsync`, `AuthorizeSensitiveReadAsync` (household-scoped — 403 when Guest lacks a `HouseholdUser` entry for that household), `AuthorizeGlobalSensitiveReadAsync` (tenant-wide — requires Member+ or App Admin, used on the cross-household `DietaryProfile` list endpoint), `RequireNonEscalatingRole`, `RequireMemberExistsAsync`. All service methods follow a consistent validation ordering: tenant context → request null-check → authorization → field normalization → existence checks → load.
- **Gathering Planning CRUD API**: Full REST controllers and services for all Gathering Planning entities, matching the Family Directory conventions (guard composition, soft-delete, tenant scoping, `AuthorizeTenantManageAsync` on writes):
  - **Events** (`/tenants/{tenantId}/events`): Full CRUD plus `POST /{eventId}/sync-plans` to manually trigger plan regeneration when the date range changes.
  - **Properties** (`/tenants/{tenantId}/properties`): Full CRUD. Unique name per tenant enforced.
  - **Accommodations** (`/tenants/{tenantId}/properties/{propertyId}/accommodations`): Full CRUD. Parent property existence checked on create; unique name per property enforced.
  - **EventAttendance** (`/tenants/{tenantId}/events/{eventId}/attendance`): GET list/GET/PUT (upsert)/DELETE. Unique constraint on `(eventId, memberId, day)`; `AuthorizeMemberEditAsync` guards writes.
  - **MealTemplates** (`/tenants/{tenantId}/events/{eventId}/meal-templates`): Full CRUD. Creating or updating a template's `MealTypes` flag **or** `StartDate`/`EndDate` sub-range triggers `PlanSyncService.SyncMealPlanAsync` to regenerate `MealPlan` rows. `StartDate`/`EndDate` are optional; when set, plans are scoped to that sub-window of the event rather than the full event date range. Validation requires both fields to be set together, `StartDate ≤ EndDate`, and the range to fall within the event's dates.
  - **TaskTemplates** (`/tenants/{tenantId}/events/{eventId}/task-templates`): Full CRUD. Same plan-sync trigger on create/`TimeSlots` change or `StartDate`/`EndDate` change via `PlanSyncService.SyncTaskPlanAsync`. Supports the same optional `StartDate`/`EndDate` sub-range with identical validation rules.
  - **MealPlans** (`/tenants/{tenantId}/events/{eventId}/meal-templates/{templateId}/plans`): GET list/GET/PUT only — lifecycle owned by `PlanSyncService`, no Create or Delete exposed via API. Update allows setting `Notes`, `IsException`, and `ExceptionReason`.
  - **TaskPlans** (`/tenants/{tenantId}/events/{eventId}/task-templates/{templateId}/plans`): Same read+update shape; update additionally tracks `Completed`.
  - **MealAttendance** (`/tenants/{tenantId}/events/{eventId}/meal-templates/{templateId}/plans/{planId}/attendance`): GET list/GET/PUT (upsert)/DELETE. Unique per `(planId, memberId)`. Tracks `AttendanceStatus` (Going/Maybe/NotGoing), `BringOwnFood`, and `Notes` — mirrors `EventAttendance` at the individual-meal level.
  - **MealIntents** (`/tenants/{tenantId}/events/{eventId}/meal-templates/{templateId}/plans/{planId}/intents`): GET list/GET/PUT (upsert)/DELETE. Unique per `(planId, memberId)`. Tracks `Volunteered: bool` — cook-volunteer flag, parallel in shape to `TaskIntent`.
  - **TaskIntents** (`/tenants/{tenantId}/events/{eventId}/task-templates/{templateId}/plans/{planId}/intents`): Same upsert shape; tracks `Volunteered`.
  - **AccommodationIntents** (`/tenants/{tenantId}/properties/{propertyId}/accommodations/{accommodationId}/intents`): Full CRUD. Unique per `(accommodationId, memberId, night)`; Create/Update/Delete use `AuthorizeMemberEditAsync`.
- **Equipment** (`/tenants/{tenantId}/equipment`): Full CRUD. Optional `PropertyId` scopes a piece of equipment to a specific property (e.g., kayaks stored at a lake house). Unique by `(TenantId, PropertyId, Name)`.
- **User invitations & JIT provisioning**: Tenant Managers invite people by email without any email-delivery infrastructure, and the invite UX is identical whether or not the invitee already exists in the external identity provider.
  - **`Invitation` entity** (`Email`, `Role`, optional `HouseholdId`/`HouseholdRole`, `Status` ∈ {Pending, Accepted, Revoked}, `InvitedByUserId`, `AcceptedByUserId`/`AcceptedAt`): inherits `AuditableEntity` (soft-delete) and carries `TenantId` for global-filter isolation. A filtered unique index `IX_Invitation_PendingPerTenantEmail` on `(TenantId, Email) WHERE Status=0 AND IsDeleted=0` enforces at most one live pending invite per email per tenant. `User` gained a nullable, indexed `Email` column populated from the verified token claim.
  - **`InvitationService`** (`/tenants/{tenantId}/invitations`, Manager+ via `AuthorizeTenantManageAsync` + `RequireNonEscalatingRole`): `CreateAsync` records a Pending invitation, or — when a `User` with that email already exists — grants membership and marks it Accepted immediately (idempotent against duplicate pending invites). `ListAsync` returns the tenant's invitations (revoked ones are soft-deleted and thus hidden); `RevokeAsync` soft-deletes + marks Revoked.
  - **`UserProvisioningService` + `POST /api/me/bootstrap`** (`MeController`, authenticated, not tenant-scoped): on first authenticated load it upserts the caller's internal `User` (creating it if absent — the JIT step), refreshes `Email`, and auto-claims any Pending invitations whose email matches the token's **verified** email. Email is only trusted when the IdP asserts `email_verified == true`; the spoofable `preferred_username` is never used for claim-matching (prevents claiming another address's invitation). For a brand-new user it pre-seeds the resolved id into `HttpContext.Items` so the auditing interceptor can stamp the self-created row. The Nuxt `tenant.global.ts` middleware calls bootstrap once before resolving tenants so a freshly-invited user's membership already exists when `/tenants` loads.
  - **`MembershipGrant.GrantAsync`**: shared, idempotent helper (adds `TenantUser` + optional `HouseholdUser` only if absent, never downgrading) used by both the immediate-accept and bootstrap-claim paths to keep the grant semantics in one place.
- **Event reports**: Read-only aggregate rooted under a `Reports` controller so future non-event report types slot in without restructuring. `EventReportService.GetEventMealReportAsync` (`GET /tenants/{tenantId}/reports/events/{eventId}`) joins `MealAttendance` through `MealPlan`→`MealTemplate` plus per-day `EventAttendance`, and emits day-by-day headcounts and per-meal `going/maybe/notGoing/bringOwnFood` counts with an aggregated, case-insensitive dietary tally (folding member `DietaryTags` + `DietaryProfile` preferred-diet/allergies/restrictions) and the attendee list. Guarded by `AuthorizeGlobalSensitiveReadAsync` (Member+): members occasionally cook behind a `TaskPlan`, and aggregated dietary needs are allergy-safety information. The frontend exposes it under a top-level **Reports** nav with a dedicated, print-friendly page; the demo computes the same aggregate from `DemoStore`.
- **Optional matching task from a meal**: `CreateMealTemplateRequest.CreateMatchingTaskTemplate` (default false). When set, `MealTemplateService` creates a standalone `TaskTemplate` mirroring the meal (same name, with a `" (cook)"` suffix on conflict; same date sub-range; meal types mapped to time slots Breakfast→Morning, Lunch→Midday, Dinner→Evening, falling back to Anytime) and runs `PlanSyncService.SyncTaskPlanAsync` so the task plans align day-for-day with the meal plans. No FK links the two aggregates — the mapping is a create-time convenience. The demo repository mirrors this via a shared `mealTypeFlagsToTaskSlotFlags` helper.
- **Frontend create/edit UI (MVP completion)**: Wired the previously API-only entities into pages/modals following the established repository-composable pattern — member create page; household create + edit via the unified `GsHouseholdModal`; property edit + delete; event create + edit (with delete); and meal/task template create/edit modals (`GsMealTemplateModal`/`GsTaskTemplateModal`, sharing `GsTemplateDateRangeField`) surfaced on the event detail page alongside a Meals tab and per-template manage actions. All actions return a success boolean so modals keep the user's input on validation failure; the invite flow uses `GsInviteUserModal` on the user-management page with a pending-invitations section.
- **Custom extensible attributes**: All major entities (`Tenant`, `Property`, `Accommodation`, `Equipment`, `Event`, `MealTemplate`, `TaskTemplate`, `Household`, `HouseholdMember`) support role-gated key-value metadata via a unified attribute pattern:
  - **Visibility**: Each attribute stores a `TenantMinRole` byte — callers only see attributes for which their tenant role is at or above the threshold. `Household` and `HouseholdMember` attributes additionally support a `HouseholdMinRole` bypass so household-private data is accessible to household managers regardless of tenant role.
  - **API shape**: Attributes are embedded in the parent entity DTO. Single-entity GET returns visible attributes; list endpoints return `attributes: []`. Parent entity PUT accepts an optional `attributes` array with full-replace semantics — supplying it replaces all visible attributes while preserving hidden ones; omitting it leaves attributes untouched.
  - **`AttributeSyncHelper.SyncAsync`**: Shared service helper used by all 9 entity services. Loads existing rows with `IgnoreQueryFilters()` to surface soft-deleted entries for re-activation, then diffs and saves in a single `SaveChanges` call to avoid unique-index violations on `(TenantId, parentId, Key)`.
  - **Frontend**: `AttributeEntry` (read shape — `id, key, value, tenantMinRole, householdMinRole`) and `AttributeWriteEntry` (write shape) types in `types.ts`. All parent entity repository create/update methods accept an optional `attributes` parameter; live repos pass it in the request body; demo repos embed it on the in-memory entity object with full-replace on update.
- **PlanSyncService**: Extracted from `EventService` into a shared injectable service (`AddScoped<PlanSyncService>()`). Owns `SyncEventPlansAsync` (full event date-range sync), `SyncTaskPlanAsync`, and `SyncMealPlanAsync`. Both `EventService` (date-change sync) and template services (create/update sync when `MealTypes`/`TimeSlots` or `StartDate`/`EndDate` changes) call it without duplication.
- **Tenant-manage authorization**: `IMemberAuthorizationService.CanManageTenantAsync` returns true for App Admins or tenant Owner/Manager roles. `ServiceGuards.AuthorizeTenantManageAsync` wraps it and is applied to all write operations on event-management entities (Events, Properties, Accommodations, Templates, Plans).
- **Unified attendance UX (Option E — Cell + Modal)**: The event attendance grid now integrates day attendance and meal attendance into a single view, eliminating the need for a separate Meals tab.
  - **`GsDayAttendanceCell.vue`** — Replaces the plain `GsAttendanceToggle` in each grid cell. Shows the day `AttendanceStatus` icon prominently plus compact B/L/D sub-indicators for each meal plan that exists on that day. Clicking opens the detail modal.
  - **`GsDayAttendanceModal.vue`** — Modal for a single member × day: set day attendance and per-meal attendance in one place. Meal controls are visually disabled (and ignored) when day status is `NotGoing`.
  - **`GsDayAttendanceBulkModal.vue`** — Follow-up modal shown after a bulk row or column set (when status is not `NotGoing`). Lets the user optionally bulk-apply meal attendance for the same scope — either a single status for all meal types, or per-meal-type individually.
  - **`GsEventAttendanceGrid.vue`** overhaul: integrates `useEventMealData`, passes meal data to `GsDayAttendanceCell` and `GsDayAttendanceModal` via the new `#cell` slot, wires `GsDayAttendanceBulkModal` after bulk row/column actions. Setting a day to `NotGoing` auto-cascades to `NotGoing` for all meal plans on that day (no prompt — logical consequence).
  - **`GsAttendanceGrid.vue`** gains an optional `hint` prop and a `#cell` slot, allowing consumers to inject a custom cell component in place of the default `GsAttendanceToggle`. Responsive max-height and per-cell borders also improved.
  - **`useEventMealData` composable**: Fetches all `MealPlan` records across every template for an event, then loads `MealAttendance` per plan. Exposes `mealPlansByDay` (plans grouped and sorted by day), `getAttendance(planId, memberId)`, and `upsert(planId, householdId, memberId, status)`. Meal data loads asynchronously so it does not block the initial grid paint.
  - **Template section date-range labels**: `GsMealTemplateSection` and `GsTaskTemplateSection` now display a formatted date-range badge (e.g., "Mon Jun 2 – Wed Jun 4") next to the template name when `startDate`/`endDate` are set.

## Planned Enhancements
- **Email/Graph invitation delivery**: Layer out-of-band invite delivery (Entra Graph invite or email) on top of the existing app-managed `Invitation` model — the data model and JIT claim flow are designed so this slots in without rework.
- **Task sign-up flows**: Capacity enforcement using `TaskTemplate.MinimumAssignees` — surface unfilled slots and prevent overbooking when members volunteer via `TaskIntent`.
- **Accommodation arbitration**: Surface arbitration metadata through dedicated admin endpoints so accommodation conflicts can be resolved with transparent priority rules and audit trails.
- **Household migration workflow**: When members age out of a parent household, allow Tenant Owner/Manager to create a new household and migrate the member, transferring or creating their `HouseholdUser` entry in the new household.
- **Row-Level Security**: Evaluate SQL Server Row-Level Security (RLS) as a defense-in-depth backstop to the existing application-layer tenancy scoping.

## Architecture Direction
Treat households and events as separate aggregates linked through member IDs, enabling independent evolution of directory data and event participation while maintaining traceability.

- **API and workflow alignment**: Both Family Directory and Gathering Planning bounded contexts are now fully exposed through REST controllers with consistent guard composition, tenant-scoped authorization, and soft-delete conventions. The full entity graph — from `Tenant` through `Household`/`HouseholdMember` to `Event`, `MealTemplate`/`MealAttendance`/`MealIntent`/`TaskPlan`/intents, and `Property`/`Accommodation`/`AccommodationIntent` — is accessible via the API, with matching create/edit UI for the MVP surface. Aggregate reporting has begun with the event meal/attendance report under the `Reports` root (daily headcounts + per-meal dietary tallies); next focus: task-coverage-per-slot reports and enforcement of capacity constraints (`TaskTemplate.MinimumAssignees`).
- **Authorization refinement**: Four-tier authorization (App Admin → Tenant Role → Event Role → Profile/Intent Role) is fully in place. `CanManageEventAsync` (Coordinator+) gates event resources; `CanAssignIntentForMemberAsync` (Coordinator+/Self/Household Manager) gates intent and attendance records; `CanManageTenantAsync` (Manager+) gates structural resources. `GetSensitiveReadScopeAsync` enforces per-household sensitive field visibility via `SensitiveReadScope` (`Global`/`ForHouseholds`/`None`). Next steps: proper HTTP status differentiation (401 vs 403), TenantUser/HouseholdUser role-management endpoints (guarded by `RequireNonEscalatingRole` and `HasOtherOwnerAsync`).
