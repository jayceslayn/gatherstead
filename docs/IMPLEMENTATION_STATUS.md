# Implementation Status and Next Steps

## Implemented Features
- **Shared foundation**: Tenants own households, properties, events, and users to keep families isolated.
- **Family directory context**: Households group members; member records store names, birth dates, dietary notes/tags.
- **Gathering planning context**: Events tie to properties and manage meal templates/plans, chore templates/plans, accommodations, and member intents for attendance, meals, stays, and chores.
- **Detailed implementation**: Multi-tenant backbone with auditing, households and members (including dietary metadata), relationship graph, contact/addresses/attributes, dietary profiles, event scaffolding with meal templates (`MealTemplate` → `MealPlan` → `MealAttendance` + `MealIntent`) and chore templates (`ChoreTemplate` → `ChorePlan` → `ChoreIntent`), daily attendance, accommodation intents with arbitration metadata, and tenant identifiers on dependent tables to enforce tenant-aware filters across the graph.
- **Family Directory CRUD API**: Full CRUD controllers and services for all Family Directory sub-entities:
  - **Addresses** (`/members/{memberId}/addresses`): CRUD with automatic IsPrimary flag management (setting one primary unsets others).
  - **Contact Methods** (`/members/{memberId}/contacts`): CRUD for email/phone/other with same IsPrimary logic.
  - **Member Attributes** (`/members/{memberId}/attributes`): CRUD for extensible key-value pairs with duplicate-key validation.
  - **Member Relationships** (`/members/{memberId}/relationships`): CRUD with cross-member validation (related member must exist in the same tenant, can span households), self-relationship guard, and duplicate relationship check.
  - **Dietary Profiles** (`/members/{memberId}/dietary-profile`): GET/PUT(upsert)/DELETE for one-to-one profiles, plus a tenant-level list endpoint (`/tenants/{tenantId}/dietary-profiles?memberIds=`) for aggregating dietary needs across attending members for meal planning.
- **JWT Bearer authentication (Entra External ID / Azure AD B2C)**: Authentication uses standard JWT Bearer tokens validated against a Microsoft external identity provider (Entra External ID or Azure AD B2C). The `ExternalIdentity` configuration section supports both providers — Entra External ID uses `ciamlogin.com` endpoints while B2C uses `b2clogin.com`; the authority URL conditionally includes the policy segment only when `SignUpSignInPolicyId` is configured. The API validates issuer, audience, lifetime, and signing keys via OIDC metadata discovery. Token revocation is supported via a `RevokedTokens` table checked in the `OnTokenValidated` event. `HttpContextCurrentUserContext` resolves the external identity provider subject (`sub` claim) to an internal `User.Id` via a DB lookup, with per-request caching in `HttpContext.Items`. (Note: Consider PASETO migration when ecosystem support improves.)
- **Rate limiting**: Sliding-window rate limiter configured in the API pipeline to enforce request throttling per the secure-defaults principle.
- **Tenant-scoped authorization**: `RequireTenantAccessAttribute` authorization filter that extracts the tenant from the route, verifies the user's `TenantUser` membership, and enforces a role hierarchy (Owner > Manager > Member > Guest). Applied to tenant-aware controllers.
- **Always Encrypted data protection**: Shifted from application-level encryption to SQL Server Always Encrypted, delegating column-level encryption of sensitive fields to the database driver rather than custom application code.
- **Project restructure**: Migrated data projects from `packages/db/` to `src/Gatherstead.Data/` and `src/Gatherstead.Data.Setup/`, aligning namespaces under `Gatherstead.Data`.
- **Framework upgrade**: Updated all projects to target .NET 10 (`net10.0`) with current package references.
- **Composable soft-delete query filters**: Global query filters in `GathersteadDbContext` now conditionally apply the soft-delete clause via a `_includeDeleted` field, while tenant isolation remains unconditional. List/read endpoints accept `?includeDeleted=true`, RBAC-gated to `TenantRole.Manager+` in `RequireTenantAccessAttribute`. The authorization decision flows through `IIncludeDeletedContext` → `HttpContext.Items`, ensuring the raw query parameter cannot bypass role checks. Lower-role users' flag is silently ignored.
- **Cross-tenant write prevention**: The `AuditingSaveChangesInterceptor` validates that every new entity's `TenantId` matches the current tenant context before saving, throwing an `InvalidOperationException` on mismatch. This provides a defense-in-depth backstop against accidental or malicious cross-tenant writes.
- **Temporal history retention**: `Gatherstead.Data.Setup` configures a 1-year retention policy on all temporal (system-versioned) tables, keeping history bounded and storage predictable.
- **Integration test suite**: A dedicated `Gatherstead.Api.Tests` project covers the authentication pipeline (JWT Bearer validation, invalid/expired/wrong-key tokens), CORS policy, rate limiting, security headers, tenant access authorization, token revocation, and context propagation (`ICurrentTenantContext`, `ICurrentUserContext`, `IIncludeDeletedContext`).
- **Supply-chain & dependency policy**: Committed a tiered stand-off policy for dependency updates ([SECURITY-DEPS.md](SECURITY-DEPS.md)) with CISA-KEV-based exception criteria and an emergency-patch runbook. Wired Dependabot ([.github/dependabot.yml](../.github/dependabot.yml)) for weekly grouped PRs across nuget/npm/github-actions. Added CI gates to [.github/workflows/build-and-test.yml](../.github/workflows/build-and-test.yml): `audit-nuget`, `audit-pnpm` (`--audit-level=high`), and `dependency-review`. All .NET projects (API, Data, Data.Setup, Api.Tests) set `RestorePackagesWithLockFile=true` with committed `packages.lock.json`; CI builds use `--locked-mode` / `--frozen-lockfile` for integrity.
- **Resource-level RBAC (Self, Household Admin)**: Fine-grained write authorization enforced at the service layer via `IMemberAuthorizationService`. The authorization decision tree evaluates (in order): tenant Owner/Manager override, Self (user's own linked `HouseholdMember` record), and Household Admin (`HouseholdRole.Admin`). Read access remains open to all tenant members. Key changes:
  - `HouseholdMember` gains a nullable `UserId` FK linking to `User` (one-to-many: one user can be linked to multiple members across households) and a `HouseholdRole` enum (`Admin`, `Member`).
  - `User` gains a `HouseholdMembers` navigation collection.
  - All Family Directory write services (`HouseholdService`, `HouseholdMemberService`, `AddressService`, `ContactMethodService`, `DietaryProfileService`, `MemberAttributeService`, `MemberRelationshipService`) enforce `CanEditMemberAsync` or `CanManageHouseholdAsync` before mutations.
  - `CreateHouseholdMemberRequest` and `UpdateHouseholdMemberRequest` accept an optional `UserId` for linking members to authenticated users. Only Tenant Owner/Manager or Household Admin can set `UserId`; the field is rejected for non-privileged users to prevent privilege escalation.
  - Per-request caching of tenant role and linked member data via `HttpContext.Items` avoids redundant DB queries within a single request.
- **Tenant Owner-only mutations**: Tenant update and delete operations are restricted to the `TenantRole.Owner` role.
- **App Admin role**: Platform-level `IsAppAdmin` flag on the `User` entity grants full administrative privileges. App Admins bypass all tenant-level (`RequireTenantAccessAttribute`) and resource-level (`IMemberAuthorizationService`) authorization checks. Key behaviors:
  - Tenant creation (`POST /api/tenants`) is restricted exclusively to App Admins via `RequireAppAdminAttribute`. The request specifies an `OwnerUserId` to designate the initial tenant Owner.
  - `GET /api/tenants` returns all tenants for App Admins, while regular users see only tenants they belong to.
  - App Admin status is resolved per-request via `IAppAdminContext` with `HttpContext.Items` caching.
  - The first App Admin must be set via direct SQL (`UPDATE Users SET IsAppAdmin = 1 WHERE ...`).
- **Localization (i18n)**: `@nuxtjs/i18n` v10 integrated with lazy-loaded locale files (English, Spanish), `prefix_except_default` URL strategy, cookie persistence, and browser language detection. See [LOCALIZATION.md](agents/plans/LOCALIZATION.md).
- **Demo site**: `demoMode` runtime config flag implemented; `useAuth`, `useTenants`, and route middleware all branch on it, stubbing auth and tenant data with in-memory values. Full localStorage persistence layer is implemented via `DemoStore` (reactive `ref()` arrays persisted to `localStorage`). Demo entities use `crypto.randomUUID()` for IDs. `seedDemoData.ts` programmatically creates all seed data through repository CRUD methods in dependency order, exercising the same code paths as real usage. Demo repositories auto-generate `MealPlan`/`ChorePlan` records per `(day × mealType/timeSlot)` when a template is created (matching the live `PlanSyncService` behavior). A "Reset Demo Data" button in the demo limits modal calls `clearDemoStore()` and re-runs `seedDemoData`. See [DEMO_SITE.md](agents/plans/DEMO_SITE.md).
- **Hybrid rendering**: Nuxt `routeRules` configured — public root is prerendered for SEO; authenticated tenant routes (`/tenants/**`) are client-only SPA.
- **Event plan auto-generation**: `PlanGenerator` computes `ChorePlan` and `MealPlan` diffs when an event's date range changes — adding plans for new days, restoring soft-deleted non-exception plans, and pruning empty plans for removed days. `EventService.SyncPlansAsync()` is the call site, covered by unit tests.
- **Guard composition pattern**: Static `ServiceGuards` class provides composable async guards (`RequireRequest`, `LoadOrNotFoundAsync`, `AuthorizeTenantManageAsync`, `AuthorizeMemberEditAsync`, `AuthorizeHouseholdManageAsync`, `RequireMemberExistsAsync`) that short-circuit on failure by mutating the shared response object. All service methods follow a consistent validation ordering: tenant context → request null-check → authorization → field normalization → existence checks → load.
- **Gathering Planning CRUD API**: Full REST controllers and services for all Gathering Planning entities, matching the Family Directory conventions (guard composition, soft-delete, tenant scoping, `AuthorizeTenantManageAsync` on writes):
  - **Events** (`/tenants/{tenantId}/events`): Full CRUD plus `POST /{eventId}/sync-plans` to manually trigger plan regeneration when the date range changes.
  - **Properties** (`/tenants/{tenantId}/properties`): Full CRUD. Unique name per tenant enforced.
  - **Accommodations** (`/tenants/{tenantId}/properties/{propertyId}/accommodations`): Full CRUD. Parent property existence checked on create; unique name per property enforced.
  - **EventAttendance** (`/tenants/{tenantId}/events/{eventId}/attendance`): GET list/GET/PUT (upsert)/DELETE. Unique constraint on `(eventId, memberId, day)`; `AuthorizeMemberEditAsync` guards writes.
  - **MealTemplates** (`/tenants/{tenantId}/events/{eventId}/meal-templates`): Full CRUD. Creating or updating a template's `MealTypes` flag triggers `PlanSyncService.SyncMealPlanAsync` to regenerate `MealPlan` rows for the event's date range.
  - **ChoreTemplates** (`/tenants/{tenantId}/events/{eventId}/chore-templates`): Full CRUD. Same plan-sync trigger on create/`TimeSlots` change via `PlanSyncService.SyncChorePlanAsync`.
  - **MealPlans** (`/tenants/{tenantId}/events/{eventId}/meal-templates/{templateId}/plans`): GET list/GET/PUT only — lifecycle owned by `PlanSyncService`, no Create or Delete exposed via API. Update allows setting `Notes`, `IsException`, and `ExceptionReason`.
  - **ChorePlans** (`/tenants/{tenantId}/events/{eventId}/chore-templates/{templateId}/plans`): Same read+update shape; update additionally tracks `Completed`.
  - **MealAttendance** (`/tenants/{tenantId}/events/{eventId}/meal-templates/{templateId}/plans/{planId}/attendance`): GET list/GET/PUT (upsert)/DELETE. Unique per `(planId, memberId)`. Tracks `AttendanceStatus` (Going/Maybe/NotGoing), `BringOwnFood`, and `Notes` — mirrors `EventAttendance` at the individual-meal level.
  - **MealIntents** (`/tenants/{tenantId}/events/{eventId}/meal-templates/{templateId}/plans/{planId}/intents`): GET list/GET/PUT (upsert)/DELETE. Unique per `(planId, memberId)`. Tracks `Volunteered: bool` — cook-volunteer flag, parallel in shape to `ChoreIntent`.
  - **ChoreIntents** (`/tenants/{tenantId}/events/{eventId}/chore-templates/{templateId}/plans/{planId}/intents`): Same upsert shape; tracks `Volunteered`.
  - **AccommodationIntents** (`/tenants/{tenantId}/properties/{propertyId}/accommodations/{accommodationId}/intents`): Full CRUD. Unique per `(accommodationId, memberId, night)`; Create/Update/Delete use `AuthorizeMemberEditAsync`.
- **PlanSyncService**: Extracted from `EventService` into a shared injectable service (`AddScoped<PlanSyncService>()`). Owns `SyncEventPlansAsync` (full event date-range sync), `SyncChorePlanAsync`, and `SyncMealPlanAsync`. Both `EventService` (date-change sync) and template services (create/update sync) call it without duplication.
- **Tenant-manage authorization**: `IMemberAuthorizationService.CanManageTenantAsync` returns true for App Admins or tenant Owner/Manager roles. `ServiceGuards.AuthorizeTenantManageAsync` wraps it and is applied to all write operations on event-management entities (Events, Properties, Accommodations, Templates, Plans).

## Planned Enhancements
- **Daily attendance summaries**: Aggregate per-day headcounts from attendance intents so event planners can see who is present on which days at a glance.
- **Chore sign-up flows**: Capacity enforcement using `ChoreTemplate.MinimumAssignees` — surface unfilled slots and prevent overbooking when members volunteer via `ChoreIntent`.
- **Accommodation arbitration**: Surface arbitration metadata through dedicated admin endpoints so accommodation conflicts can be resolved with transparent priority rules and audit trails.
- **Household migration workflow**: When members age out of a parent household, allow Tenant Owner/Manager to create a new household and migrate the member, automatically assigning them the Household Admin role.
- **Row-Level Security**: Evaluate SQL Server Row-Level Security (RLS) as a defense-in-depth backstop to the existing application-layer tenancy scoping.

## Architecture Direction
Treat households and events as separate aggregates linked through member IDs, enabling independent evolution of directory data and event participation while maintaining traceability.

- **API and workflow alignment**: Both Family Directory and Gathering Planning bounded contexts are now fully exposed through REST controllers with consistent guard composition, tenant-scoped authorization, and soft-delete conventions. The full entity graph — from `Tenant` through `Household`/`HouseholdMember` to `Event`, `MealTemplate`/`MealAttendance`/`MealIntent`/`ChorePlan`/intents, and `Property`/`Accommodation`/`AccommodationIntent` — is accessible via the API. Next focus: surface aggregate/summary endpoints (daily attendance headcounts, chore coverage per slot) and enforcement of capacity constraints (`ChoreTemplate.MinimumAssignees`).
- **Authorization refinement**: Three-tier authorization (App Admin → Tenant Role → Resource Role) is fully in place, including the `CanManageTenantAsync` tier for event-management entities. Next steps include proper HTTP status differentiation (401 for authentication failures, 403 for insufficient permissions) and unit/integration tests for the authorization decision trees.
