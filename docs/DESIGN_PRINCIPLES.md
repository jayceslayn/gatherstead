# Design and Security Principles

- **Privacy by design**: Treat personal data (PII, contact details, dietary notes) as sensitive by default. Favor minimization, avoid unnecessary replication across contexts, and prefer references over denormalized copies.
- **Tenant isolation**: Enforce tenant boundaries in persistence and APIs; every query should filter by tenant, and indexes should include tenant keys to avoid cross-family data leakage.
- **Data protection in transit and at rest**: Require TLS for all client/server and service/service traffic. Use encryption-at-rest for databases and leverage SQL Server Always Encrypted with Secure Enclaves for high-sensitivity columns (e.g., contact methods, medical/dietary notes), keeping encryption transparent to the application via the database driver.
- **Role-aware access control**: Permissions are enforced at four tiers.
  - **Platform-level**: `User.IsAppAdmin` grants full administrative access — App Admins bypass all other authorization, can create tenants (specifying the initial Owner), and see all tenants in list endpoints.
  - **Tenant-level**: `TenantRole` (Owner=0 > Manager=1 > Coordinator=2 > Member=3 > Guest=4) governs broad operations. Owner alone can update or delete the tenant itself. Manager+ can manage structural resources (Properties, Accommodations, Households, HouseholdMembers, Users). Coordinator+ can manage event resources (Events, Templates, Plans). Member+ can read sensitive member data across the entire tenant; Guest read access is scoped by their `HouseholdUser` entries (see sensitive read scope below).
  - **Event-assignment-level**: `IMemberAuthorizationService.CanAssignIntentForMemberAsync` controls write access to intent and attendance records (EventAttendance, MealAttendance, MealIntent, TaskIntent, AccommodationIntent, and future EquipmentIntent) — permits App Admin, tenant Owner/Manager/Coordinator, Self (user's linked member within this tenant via `TenantUser.LinkedMemberId`), or Household Manager.
  - **Profile-level**: `IMemberAuthorizationService.CanEditMemberAsync` controls write access to member profile data (name, contact info, addresses, dietary notes, attributes, relationships) — permits App Admin, tenant Owner/Manager, Self, or Household Manager (via `HouseholdUser.Role`). Household Managers can manage all members within their household. The Self link (`TenantUser.LinkedMemberId → HouseholdMember`) is per-tenant, so users who belong to multiple tenants each have a distinct self-profile in each; setting or changing this link is restricted to Tenant Owner/Manager or Household Manager. Role assignment is a Tenant Manager+ operation — a user may only grant roles at or below their own level; Household Managers cannot independently assign roles. A tenant must always retain at least one Owner. Sensitive capabilities such as querying soft-deleted entities require Manager+ and are silently unavailable to lower-privilege users.
  - **Sensitive read scope**: `IMemberAuthorizationService.GetSensitiveReadScopeAsync` returns a `SensitiveReadScope` — `Global` (App Admin or TenantRole ≤ Member), `ForHouseholds(ids)` (Guest with one or more `HouseholdUser` entries — scoped to those households), or `None` (Guest with no `HouseholdUser` entries). `HouseholdMemberDto` nulls `BirthDate`, `DietaryNotes`, and `DietaryTags` when the caller lacks sensitive read access for that household; sub-entity endpoints (Addresses, ContactMethods, MemberAttributes, MemberRelationships, DietaryProfile) return 403.
- **Verified-identity invitation claiming**: App-managed invitations are matched to users by email, and accepting one grants real tenant membership — so the email must be trustworthy. `UserProvisioningService` only claims invitations against an email the identity provider marks verified (`email_verified == true`); it never matches on `preferred_username` or any self-asserted claim, preventing a user from claiming an invitation intended for someone else's address and escalating into a tenant. Invitation creation additionally enforces non-escalation (`RequireNonEscalatingRole`) so an inviter cannot grant a role above their own.
- **Auditability and accountability**: Capture who created/updated sensitive records (members, contacts, intents, assignments) and when; surface audit trails in admin tooling to detect misuse. Soft-deleted entities are retained with full audit metadata (who deleted, when) and can be queried by authorized users for review or restoration.
- **Soft-delete by default**: All auditable entities use soft-delete rather than hard-delete. Global query filters exclude deleted records from all queries unless explicitly overridden by an authorized request. This preserves data integrity, supports undo workflows, and maintains referential consistency across the entity graph.
- **Secure defaults and hardening**: Apply input validation, rate limiting, CSRF protections, and strong authentication. Disable risky defaults (e.g., open CORS, overly broad tokens) and maintain secret rotation and key management practices.
- **Data lifecycle and consent**: Track consent for sharing details, support export/delete workflows per tenant requests, and document retention policies for logs and backups.
- **Localization-ready UI**: All user-visible text in the web app must go through the i18n system (`@nuxtjs/i18n`). Never hardcode strings in templates or components. This ensures the app can be translated to additional languages without code changes, and keeps date/number/currency formatting locale-aware. See [LOCALIZATION.md](agents/plans/LOCALIZATION.md) for conventions and implementation details.
- **Consistent, accessible UI**: The web UI follows a shared set of presentation and interaction conventions — mobile-first layouts, a centralized status icon/color language (color never the sole differentiator), reusable `Gs*` components, confirm-before-delete, and modals that close only on success. These are mandatory for every page and component. See [WEB-UI-DESIGN.md](WEB-UI-DESIGN.md).
- **Operational readiness**: Establish monitoring for anomalous access patterns, failed logins, and data exfiltration signals. Include security reviews in change management and keep dependencies updated with vulnerability scanning.
- **Supply-chain hygiene**: Dependencies follow a tiered stand-off policy — routine updates wait long enough for the community to flag malicious publishes, but actively-exploited CVEs (e.g. CISA KEV listings) are patched immediately. Every PR runs `audit-nuget`, `audit-pnpm`, and `dependency-review` gates. NuGet and pnpm lockfiles are committed and enforced via `--locked-mode` / `--frozen-lockfile`, and pnpm's `onlyBuiltDependencies` allowlist limits postinstall-script blast radius. See [SECURITY-DEPS.md](SECURITY-DEPS.md) for the full policy, tiers, and emergency-patch runbook.
